# HALMark v0.9.12-draft — March 2026 Release Notes

Three new Candidate Footguns contributed by Caitlin (Anthropic, Sonnet 4.6) from production-confirmed failures on Home Assistant 2026.3.0 in a live ZenOS-AI install. All three are type_safety failures in the Jinja template sandbox. All were silent at authoring time — no syntax error, no linting signal, runtime-only failure.

---

## New Footguns

FG-32 through FG-34 are all `Candidate` status, pending board ratification.

**FG-32 — Trigger Object Serialization Failure** `hard_fail`
Attempting to pass the trigger object directly to `tojson` — a common LLM debugging suggestion — throws a TemplateError at render time because the trigger object contains datetime fields that are not JSON-serializable. The correct pattern is to destructure the trigger manually, extracting only primitive fields (id, platform, entity_id). `trigger.to_state` and `trigger.from_state` are also non-serializable. This is the kind of silent-looking failure that only surfaces when the automation fires in production. Proposed arena placement: Arena 0 (The Gauntlet) — trigger object type sense is fundamental.

**FG-33 — Python repr Rendering Trap** `soft_fail`
When a dict or list variable is interpolated into a string context without the `tojson` filter, HA renders it using Python's `repr()` format — not JSON. The divergence is: `None` → `'None'`, `True/False` → `'True'/'False'`, single-quoted strings. Any downstream system expecting valid JSON silently receives malformed input. `from_json` will fail. Write-path counterpart to FG-12 (which is read-path). Proposed arena placement: Arena 1 (Template Correctness).

**FG-34 — variables: Block Auto-Deserialization** `soft_fail`
In HA's `variables:` block, when a template renders to a valid JSON string (e.g. via `| tojson`), HA automatically deserializes it back to a native Python object before the variable is stored. A downstream `from_json` call on that variable then throws: "got dict, expected string." The correct approach is to access the variable directly — no `from_json` needed. The variables: block is a typed store, not a string store. Variables-context-specific variant of FG-12. Proposed arena placement: Arena 1 (Template Correctness).
