# HALMark v0.9.13-draft — March 2026 Release Notes

Five new Candidate Footguns across two contributor batches. All production-confirmed on Home Assistant 2026.3.0 in a live ZenOS-AI install. Sources: systemtools 4.1.0 reload implementation review, dojotools_labels SP1 analysis, and FileCabinet drawer read pattern audit.

---

## New Footguns

FG-35 through FG-39 are all `Candidate` status, pending board ratification.

**FG-35 — automation.reload Silently Kills In-Flight Automations** `hard_fail` `deference_required`
`automation.reload` terminates all currently running automation action sequences immediately — no error, no event, no recovery path. An automation mid-sequence (waiting on a delay, running a long script) is destroyed silently. Agent must warn the user and require confirmation before calling. Prefer `homeassistant.reload_all` for routine post-edit reloads.

**FG-36 — homeassistant.reload_all Silent Abort on Config Check Failure** `hard_fail`
`reload_all` runs an internal config check before reloading. If that check fails, reload_all aborts silently — no exception, no return value, no event. An agent that calls reload_all and reports "reloaded successfully" may have reported success for a complete no-op, leaving the user with a false mental model of system state. Gate on an explicit `ha_config_check` call first.

**FG-37 — Partial Reload State from Incorrect Surgical Reload Sequencing** `soft_fail`
Calling `script.reload` followed by `automation.reload` in sequence after a multi-domain YAML edit leaves a window where scripts are live but automations reference old script definitions. Use `reload_all` for any multi-domain change. Reserve surgical reloads for confirmed single-domain edits only.

**FG-38 — FileCabinet Drawer .value Fields May Be JSON-Encoded Strings, Not Native Dicts** `hard_fail`
Cabinet drawer `.value` fields are not guaranteed to be native dicts. Write-path inconsistency means some paths store native dicts (via `variables:` block) and others store JSON strings (via `| tojson`). Calling `.get()` on a string crashes. The double-encoded variant (write path applied `| tojson` before cabinet write) requires two `from_json` passes — one is insufficient. Canonical pattern is three-round normalization with a final `is mapping` guard. Two-round pattern is now retired as the baseline — it silently returns `{}` on double-encoded drawers rather than crashing, masking real data loss.

**FG-39 — Non-Atomic Destructive Sequence: Delete Before Recreate Is Confirmed Viable** `hard_fail` `deference_required`
HA scripts have no rollback. A delete→recreate→retag sequence that fails mid-way leaves state permanently altered — delete succeeded, recreate didn't, data is gone. The invariant: at no point should both old and new be absent simultaneously. Validate all inputs and pre-confirm createability before executing the destructive step.
