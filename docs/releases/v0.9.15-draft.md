## New Footguns

**FG-41 — as_datetime Silent None on Unparseable Input Breaks Downstream Filter Chain** `hard_fail` `type_safety`

`as_datetime` returns `None` silently when it cannot parse its input — it does not throw, log, or warn. The footgun is in the guard: standard existence checks (`is not none`, `trim != ''`, `!= 'None'`) fire on the *input string*, not on the *output* of `as_datetime`. Any downstream filter that receives `None` (most commonly `as_timestamp`) then throws at runtime:

```
ValueError: Template error: as_timestamp got invalid input 'None'
```

Confirmed bad input examples from live kata drawers: `"21:17:19"` (time-only string written by pre-schema Monk), `"1775505678.824051"` (Unix timestamp float string). Both pass all three existence guards. Both blow up `as_timestamp`.

The canonical fix: `| default(none, true)` after `as_datetime`, before any further filter. The `true` parameter is required — without it, `default()` only catches `Undefined`, not `None`.

Production-confirmed on HA 2026.3.0: ninja_summarizer emission cooldown gate, ZenOS install. Submitted by Nyx (NYX-001, 2026-04-06).

## Arena & Test Development — Call for Contributions

Starting this cycle, we are actively framing out the HALMark arena test harness.

The spec defines five arenas (Arena 5 deferred to v2):

| Arena | Name | Status |
|---|---|---|
| 0 | Type Sense (The Gauntlet) | Scaffolded — 2 cases |
| 1 | Template Correctness | Scaffolded — 3 cases |
| 2 | YAML Integrity | Empty |
| 3 | Behavioral Preservation | Empty |
| 4 | Safety / Guardrails | Empty |

The runner (`arena/`), scoring config (`scoring/`), and validation tooling (`tools/`) are scaffolded but need implementation. We are looking for:

- **Test case submissions** — specific, reproducible prompts grounded in real HA failure modes. See `tests/arena_1/cases/` for the current format.
- **Runner implementation ideas** — how should model outputs be evaluated deterministically? What does a scoring run look like end to end?
- **Scoring mechanic feedback** — the weighted scoring schema (Correctness 40%, Regression Safety 25%, Structural Integrity 20%, Operational Discipline 15%) is defined in spec. Does it hold up against real test cases?

If you have ideas, production failure cases, or want to build, open an issue or PR. Label arena-related issues `arena-dev`.
