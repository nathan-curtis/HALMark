# HALMark v0.9.11-draft — March 2026 Release Notes

Six new Candidate Footguns land this month across two contributor batches, plus a full internal consistency audit against Home Assistant 2026.3.0.

---

## New Footguns

FG-26 through FG-31 are all `Candidate` status, pending board ratification.

**FG-26 — LLM Output Instruction Echo / Growth Bomb** `hard_fail`
The one that started as a real incident. When an AI integration co-locates prompt instructions with the output slot it writes back to a sensor attribute, the model echoes those instructions into persistent state on every run. The blob grows non-linearly, hits the recorder's 16,384-byte silent drop limit (FG-21), and the data is gone on restart with no error raised. Architectural separation of schema from state is the fix. Token cap and replace-not-append instruction are required. `deference_required`.

**FG-27 — State Function Access in `trigger_variables` Context** `hard_fail`
`states()`, `state_attr()`, `now()`, and the rest of the state machine helpers are not available in `trigger_variables` blocks. Models assume they are because the syntax looks identical to `variables:`. The failure is silent or raises at startup depending on the call. Elevated to `hard_fail` during review — the resulting automation misbehavior is difficult to diagnose.

**FG-28 — `iif()` All-Branch Evaluation Trap** `soft_fail`
`iif()` is a function call, not a control flow construct. All three arguments are fully evaluated before the return value is selected. Python-trained intuition about short-circuit conditionals does not apply. `from_json` on an unavailable sensor throws even when the condition would have routed away from that branch. When in doubt, use `{% if %}`.

**FG-29 — MQTT Discovery Schema Drift** `soft_fail`
MQTT discovery entities that declare `payload_on`/`payload_off` values mismatched against what the device actually publishes initialize correctly and then silently never update. AI agents generating discovery config must verify payload format or use `value_template` as a normalization layer.

**FG-30 — Light Temperature API Migration: Mired to Kelvin** `soft_fail`
References to `min_mireds`/`max_mireds` in custom `LightEntity` implementations risk `AttributeError` as HA removes the deprecated attributes. *Note: this entry introduces `python_custom_integration` scope — the first time HALMark has touched custom integration Python code. It is live in the spec and going to a board vote. If the scope expansion is ratified, it opens a formal Python integration track. If not, it is pulled cleanly.*

**FG-31 — Configuration Blast Radius: Flat File Growth When Packages Are Available** `soft_fail`
When HA packages are already enabled, AI should place new automations and scripts in a dedicated package file rather than appending to monolithic flat files. A mistake in `automations.yaml` can silently remove hundreds of unrelated automations — HA parses successfully and raises no error. When packages are not enabled, AI must surface the option and stop; restructuring requires explicit user approval. `deference_required`.
